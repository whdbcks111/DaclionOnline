# 패치노트

사용자에게 공개할 버전별 변경 기록은 `shared/patchNotes.ts` 한 곳이 소유한다. 서버의 `/패치노트` 명령과 클라이언트 `/patch-notes` 화면은 모두 `getPatchNotes/getPatchNote` snapshot API를 사용하므로 같은 내용을 중복 작성하지 않는다. 전체 Git 이력은 연속된 작업 묶음별로 버전이 배정되어 있으며, 첫 모험 튜토리얼 완성까지는 `v0.x.x` 베타 기록, 이후는 `v1.x.x` 정식 기록으로 구분한다.

## 사용자 진입

- 햄버거 메뉴의 `패치노트`: 버전 목록과 선택한 버전의 배포일·분류별 변경 내용을 보여주는 반응형 계층 화면을 연다. 버전 목록과 본문은 많은 베타 기록도 탐색할 수 있게 각각 스크롤된다. 화면은 게임 전역 스크롤 잠금과 독립된 `100dvh` 세로 스크롤 컨테이너를 사용한다.
- `/패치노트`: 채팅에는 최신 버전·배포일·전체 기록 개수와 `상세 보기` 버튼만 표시한다. 상세보기의 스크롤 화면에서 등록된 버전을 최신순으로 모두 확인한다.
- `/패치노트 1.0.0`: 채팅에는 선택한 버전 요약만 표시하고 실제 변경 내용은 상세보기에서 확인한다. 버전 자동완성과 선택적 `v` 접두사 입력도 지원한다.
- 정보성 명령이므로 공개/비공개 모드에 따라 채널 공개 여부가 결정된다.

## 데이터 규칙

`PatchNoteCategory`는 아래의 짧은 마커·표시명·표시 순서를 소유하는 클래스형 enum이다.

- `[+] 추가된 기능`
- `[+] 추가된 콘텐츠`
- `[/] 수정된 기능`: 개선·밸런스·오류 수정을 함께 포함한다.
- `[-] 삭제된 기능·콘텐츠`

각 버전은 다음 필드를 가진다.

- `version`: SemVer 형식의 배포 버전. 저장값은 `1.0.0`, 표시는 `v1.0.0` 형식을 사용한다.
- `releasedAt`: `YYYY-MM-DD` 형식의 배포일
- `sections`: 카테고리와 사용자 관점의 변경 항목

같은 버전의 작업은 새 객체를 중복 생성하지 않고 기존 버전의 적절한 section에 합친다. `getPatchNotes()`가 SemVer 역순 snapshot을 반환하고 내부 배열과 문자열 배열을 복사하므로 화면이나 명령이 마스터 데이터를 변경할 수 없다. 배포일은 버전을 대신하는 조회 key로 사용하지 않는다.

## 버전 경계

- 직전 작업 종료와 다음 작업 시작의 간격이 약 1시간 미만이면 기본적으로 같은 버전에 합친다.
- 서버 재시작·배포 시점처럼 실제 운영 묶음의 경계가 명확하면 1시간 이내여도 새 버전을 시작한다.
- 여러 커밋이 같은 재시작 전 작업이라면 커밋 수와 관계없이 하나의 버전으로 기록한다.
- 첫 모험 튜토리얼이 실제 장소·행동 성공을 검증하도록 완성된 `v0.68.0`까지는 베타다. 카르마 시스템부터 `v1.0.0` 정식 기록을 시작한다.
- 기존 이력을 재분류할 때는 커밋 작성 시각과 주제를 함께 확인하고, 내부 구현명 대신 당시 플레이어가 할 수 있게 된 행동과 체감 변경을 요약한다.

## 변경 완료 규칙

사용자에게 보이는 기능·콘텐츠·밸런스·오류 수정이 완료되면 같은 커밋에서 현재 작업 묶음의 패치노트를 갱신한다. 새 서버 재시작·배포 묶음을 시작하거나 직전 작업에서 약 1시간 이상 떨어졌다면 다음 버전을 추가한다. 리팩터링, 테스트만 추가, 문서 오탈자처럼 플레이 결과가 달라지지 않는 내부 변경은 항목으로 기록하지 않는다. 구현 세부 파일명이나 내부 key 대신 플레이어가 무엇을 새로 할 수 있고 무엇이 달라졌는지를 적는다.
